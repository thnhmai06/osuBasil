# API consumed by the osu! client

Everything the real osu! game client communicates with: persistent binary bancho protocol, and the HTTP `/web/osu-*.php` endpoints it calls alongside that connection. For what is stubbed, what is real, and why, see [`scope-decisions.md`](scope-decisions.md).

The **"Official osu! Bancho"** column compares against the **actual osu! Bancho server** (osu.ppy.sh) — not bancho.py (the open-source fork whose architecture this project ports, see [`CLAUDE.md`](../CLAUDE.md)). The wire protocol itself (packet IDs, field ordering) is defined by the real osu! client, so most packets below are identical whether compared to bancho.py or official Bancho; the significant differences lie in the chat-command layer (BanchoBot) and the fact that official Bancho stores full match history in browsable web pages. The **"Comparison"** column:

- ✅ Yes — behavior matches official Bancho
- ⚠️ Yes, with limitations — behavior differs, or Basil extends beyond official Bancho
- ❌ No — does not exist on official Bancho

See [`bot-commands.md`](bot-commands.md) for the BasilBot command reference, [`api-external.md`](api-external.md) for the `api.` host's external tournament tooling API (JSON/WebSocket, much easier to integrate than the endpoints here — if you are building an overlay/bracket tracker rather than a real osu! client, read that file instead).

## Where endpoints are hosted

Every endpoint below — bancho protocol, osu-web, beatmap assets — is a single ASP.NET Core application (`Basil.Web`), routed separately by **hostname, not path prefix**. `BanchoHostGroups.MapAll` (`src/Basil.Web/Routing/BanchoHostGroups.cs:33`) builds the actual host list at startup from a single config value:

```
Server:Domain      (settings.toml, next to the executable — same file for dev and deployment)
```

With that configured domain **and** `ppy.sh` hardcoded (so a production deployment behind a reverse proxy answering on the real osu! domain works without extra config), it generates:

| Host group                                         | Hostname                                                                | Route group            |
|----------------------------------------------------|-------------------------------------------------------------------------|------------------------|
| bancho protocol                                    | `c.<domain>`, `ce.<domain>`, `c4.<domain>`, `c5.<domain>`, `c6.<domain>` | `MapBanchoGroup`       |
| osu-web                                            | `osu.<domain>`                                                          | `MapOsuWebGroup`       |
| asset thumbnail/preview beatmap (old mirror redirect)| `b.<domain>`                                                           | `MapBeatmapAssetGroup` |
| avatar (local, offline pivot)                      | `a.<domain>`                                                            | `MapAvatarGroup`       |
| IRC gateway (TCP, not HTTP)                        | `irc.<domain>` (port 6667, not routed through ASP.NET)                  | `TcpIrcListener` (`BackgroundService`) |

`Program.cs:30` reads `Server:Domain` and passes it to `MapAll` — that is the single place the domain is threaded through; there is no per-route host config anywhere else.

**Where that config value lives:** `src/Basil.Web/settings.toml`, section `[Server]`, key `Domain` — same file for both `dotnet run` (local dev) and the published executable, no environment separation, no environment-variable override layer. Edit the file, restart the process. See [`run-deployment.md`](run-deployment.md).

**Network binding (which port/protocol the app actually listens on) is configured in `settings.toml` `[Server]`:**

- `Port` — Kestrel HTTPS listen port (default 443). Disables automatic port selection — the server binds exclusively here.
- `CertPath` / `CertPassword` — path to the HTTPS cert (PFX) and its password. Leave unset to use the ASP.NET Core dev cert or OS-level reverse proxy TLS.

No `--urls` or `ASPNETCORE_URLS` override — every setting comes from `settings.toml`. See [`run-deployment.md`](run-deployment.md) for the full deployment walkthrough.

## 1. Bancho protocol (host `c.`/`ce.`/`c4.`/`c5.`/`c6.`)

This is the persistent binary connection the osu! client opens when you click "log in". All communication — both login and every subsequent packet — is `POST /` on this host; **there is no other path**. The difference between "logging in" and "logged in, sending packets" is just one header: `osu-token`.

### General flow

```
┌──────────────┐                                    ┌──────────────┐
│  osu! client │─────────── POST / (no header) ────▶│ Basil server │
│              │◀────────── cho-token + packets ────│              │
│              │            (User ID, Channels...)  │              │
│              │                                    │              │
│              │─── POST / (osu-token: <token>) ───▶│              │
│              │◀── accumulated server packets ─────│  (Repeated   │
│              │    for that session                │   every      │
└──────────────┘                                    │   few ms)    │
                                                    └──────────────┘
```

1. **Login (first request, no `osu-token`)** — body is multi-line text composed by the client (`username\npassword_md5\nosu_version|utc_offset|display_city|client_hashes|pm_private`). Server (`OsuLoginUseCase`) authenticates and returns:
    - Header `cho-token: <random token>` — client must send this header on **every** subsequent request.
    - Body: initial binary packet stream (your user ID, channel list, presence of online players...).
2. **Every subsequent request (with `osu-token: <token>`)** — body is **1 or more binary client-to-server packets** (change map, send chat, ready...), read sequentially by `BanchoPacketDispatcher.DispatchAsync`, each routed to its handler by packet ID (see table below). Response body is **all accumulated outgoing packets for that session** since the last poll (new chat, others' presence changes, match updates...) — if nothing new, response is empty. This is the polling mechanism the client uses to receive real-time events, even over plain HTTP (not WebSocket).
3. **Unknown token** (e.g. server just restarted) → response is a "Server has restarted" message with a reconnect packet, telling the client to automatically re-login.

Each packet — both directions — has a common binary structure `[packet ID: u16][padding: u8][length: u32][payload: length bytes]`; payload fields vary by packet type. The detailed per-packet payload format is outside the scope of this document — the 42 packet types below list **which packets exist and what they do**, not byte-by-byte layout (that is defined by the osu! client itself, not Basil).

## 2. HTTP endpoints (`osu.`/`b.`/`a.`)

All routes are registered in one place, `src/Basil.Web/Routing/BanchoHostGroups.cs` — routing is by subdomain (not path prefix). **Every endpoint below returns text/html in the old osu! wire format (pipe `|`/newline-delimited), not JSON** — the sole exception is `POST /difficulty-rating`. The "Handled by" column (folded into "Usage") points to the actual class handling the work; "inline" means the logic lives directly in the route lambda, with no separate service class.

### 2.1 `c.`/`ce.`/`c4.`/`c5.`/`c6.` — bancho protocol

The client's realtime persistent connection. All requests are `POST /` — no other path on this host group. Detailed mechanism and packet table in section 1.

| API              | Usage                                                                                                                                                                                                                                                  | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    | Official osu! Bancho | Comparison |
|------------------|--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|----------------------|------------|
| Bancho over HTTP | `POST /` — `MapBanchoGroup` (inline) → `OsuLoginUseCase` (no `osu-token` header) or `BanchoPacketDispatcher` (with `osu-token` header)                                                                                                                | No token: body is treated as a login attempt and forwarded to `OsuLoginUseCase`, which returns `cho-token` header + initial packet stream. With token: session looked up; raw body forwarded to `BanchoPacketDispatcher.DispatchAsync`, which reads packet IDs and calls the appropriate handler from the bancho packet table above; accumulated outgoing packets are flushed as the response body. #### Example request/response details

**Leaderboard — `GET /web/osu-osz2-getscores.php`**

Required query params: `us` (username), `ha` (MD5 password), `s` (`0`/`1`, "called from editor song-select"), `v` (leaderboard type — number, see `LeaderboardType`), `c` (map md5), `f` (map filename), `m` (mode, number), `mods` (bitmask).

```
GET /web/osu-osz2-getscores.php?us=PlayerOne&ha=<md5 pass>&s=0&v=1&c=9f8c2e1a...&f=Camellia+-+Frontier.osu&m=0&mods=0
```

Response (plain text, not JSON), by case:

| Case                                               | Body                                                |
|----------------------------------------------------|-----------------------------------------------------|
| Wrong auth                                         | `401`                                               |
| Map not submitted (does not exist)                 | `-1\|false`                                         |
| Map needs update (checksum mismatch)               | `1\|false`                                          |
| Map has no leaderboard (ranked status insufficient)| `{status}\|false` (`{status}` is `RankedStatus` code) |
| Has leaderboard                                    | multiple lines — see below                          |

"Has leaderboard" body (each line separated by `\n`):

```
1|false|111222|5000|3|0|
0
Camellia - Frontier
5.42
<personal-best line, or empty>
9001|PlayerOne|987654|512|0|10|480|1|8|100|1|0|7|1|1700000000|1
9002|PlayerTwo|900000|400|2|20|400|3|4|90|0|0|8|2|1700000100|1
```

Each score line: `id|name|score|maxCombo|n50|n100|n300|nMiss|nKatu|nGeki|perfect(0/1)|mods|userId|rank|time|1`.

**Search beatmap — `GET /web/osu-search.php`**

Query: `u`, `h` (auth), `r` (ranked status, number), `q` (keyword), `m` (mode), `p` (page).

```
GET /web/osu-search.php?u=PlayerOne&h=<md5 pass>&r=0&q=frontier&m=0&p=0
```

Response (text, first line = result count — `101` means "page full, more results"; each subsequent line is one beatmap set):

```
1
5000.osz|Camellia|Frontier|MapperName|1|10.0|2024-01-01 00:00:00|5000|0|0|0|0|0|[5.42⭐] Insane {cs: 4 / od: 8 / ar: 9 / hp: 6}@0
```

**Score submission — `POST /web/osu-submit-modular-selector.php`**

Multipart form fields: `score` (encrypted score data, base64 text — same field name may also carry `.osr` file if client sends replay), `s`, `iv`, `osuver`, `pass`, `c1`, `sbk`, `bmk`, `st`, `ft`. This is an internal request created by the osu! client itself when completing a beatmap — not something you can hand-craft (payload encrypted with a per-version client key).

Response (plain text):

| Case                                                      | Body                                                                                                                             |
|-----------------------------------------------------------|----------------------------------------------------------------------------------------------------------------------------------|
| Passed, submit successful                                 | "chart" block pipe-delimited: `beatmapId:`, `chartId:beatmap`, `rankBefore:`/`rankAfter:`, `chartId:overall`, `achievements-new:`... |
| Failed                                                    | `error: no`                                                                                                                      |
| Beatmap not found                                         | `error: beatmap`                                                                                                                 |
| Player not found (rare, client auto-retries)              | empty                                                                                                                            |
| Duplicate score                                           | `error: no`                                                                                                                      |

**Song-select beatmap info — `POST /web/osu-getbeatmapinfo.php`**

Query: `u`, `h`. Request body **is real JSON** (the exception in this text-format group):

```json
{ "Filenames": ["Camellia - Frontier (MapperName) [Insane].osu"], "Ids": [] }
```

Response (text, 1 line per matching file): `{index}|{id}|{setId}|{md5}|{status}|{grade0}|{grade1}|{grade2}|{grade3}`
(4 grades correspond to 4 original modes osu!/Taiko/CTB/Mania, `"N"` if no score yet for that mode).

```
0|111222|5000|9f8c2e1a...|2|S|N|N|N
```

**Star rating — `POST /difficulty-rating`** (the only real JSON endpoint in this group)

Query: `b` (beatmap id, required), `mods` (bitmask, default `0`).

```
POST /difficulty-rating?b=111222&mods=64
```

Response:

```json
{ "beatmap_id": 111222, "mods": 64, "rating": 5.87 }
```

Note `beatmap_id` keeps snake_case (not changed to `beatmapId`) — this is a raw object with hardcoded field names in code, not going through camelCase policy like the endpoints in [`api-external.md`](api-external.md).
`404` (text) if beatmap does not exist in DB; different error text if DB has a row but the `.osu` file is missing on disk.

### 2.4 `a.` — avatar (offline pivot)

bancho.py has no local avatar storage (its `a.<domain>` host always forwards to a real CDN). This server stores flat avatar files as `{userId}.{ext}` under `Storage__AvatarsPath`.

| API                   | Usage                                       | Description                                                                                                                                                               | Official osu! Bancho                           | Comparison |
|-----------------------|---------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------|------------------------------------------------|------------|
| Serve local avatar    | `GET /{userId:int}` — `MapAvatarGroup` (inline) | serve the first file matching `{userId}.*` in `Storage__AvatarsPath` (content-type png/jpg/jpeg/gif inferred from extension); 404 if no file found | Always forwards to a real CDN instead | ❌ No      |

Example: `GET https://a.basil.example/7` → `200`, `Content-Type: image/png`, body is avatar image bytes for user id `7` (server looks for `7.png`, `7.jpg`... in order); `404` if user has no avatar.
